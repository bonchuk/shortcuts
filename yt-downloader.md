# Завантажувач з YouTube (iPhone) — «Завантажити з YouTube 2.4» + «YTSave»

Робоча пара: **Завантажити з YouTube 2.4.shortcut** + **YTSave.shortcut** (обидва в цій папці, підписані `shortcuts sign --mode anyone`). Працюють тільки разом. **Статус: РОБОЧА, підтверджено на телефоні 2026-07-21.** Вивід yt-dlp іде прямо на екран a-Shell (без перенаправлення в лог — див. граблі нижче), тому прогрес і помилки видно наживо.

Сценарій: YouTube → Поділитися → «Завантажити з YouTube» → відкривається a-Shell і видно прогрес завантаження → по завершенню сам відкривається другий шорткат і кладе відео в галерею в H.264 mp4 — CapCut відкриває без перекодування. Замінює ланцюжок «4K Video Downloader + Shutter Encoder» з ПК-інструкції.

Якість: **до 1080p**. Це стеля H.264 у YouTube — 4K він віддає тільки у VP9/AV1, які монтажні програми не їдять (саме тому в ПК-інструкції було перекодування). Для Reels/TikTok 1080p достатньо.

## Чому два шорткати (історія v1.0)

v1.0 качала через a-Shell Execute Command в extension-режимі — і падала з «The operation took too long to complete»: iOS дає розширенню обмежений бюджет часу, 1-хвилинне відео проскакувало, довші — ні. Підтверджено на телефоні 2026-07-20.

v2.0/2.1 обходили ліміт через URL-схему `ashell://<encoded command>` (Text → URL Encode → Open URL) — НЕ ПРАЦЮЄ з шитера «Поділитися»: a-Shell блимає і вилітає на домашній екран, команда не виконується взагалі (лог не пишеться, термінал порожній). Схоже, share-розширенню не дозволено повноцінно відкривати застосунок з URL-payload. Сам механізм `ashell://` при цьому справний (раунд-тріп кодування перевірено на Foundation байт-у-байт).

v2.2 запускає команду через штатний інтент a-Shell Execute Command з **openWindow=open** («Open the app: Yes») — той самий канал, який у дебагу різалки відкривав застосунок з команди шитера. Інтент не блокує і не чекає (тому таймаут розширення не застосовується), команда виконується в застосунку:

1. «Завантажити з YouTube 2.2» — єдина дія: a-Shell Execute Command (openWindow=open) з багаторядковою командою (виконується послідовно, рядок за рядком).
2. Останній рядок команди — `openurl shortcuts://run-shortcut?name=YTSave` — виконується після завершення yt-dlp і запускає другий шорткат.
3. «YTSave»: Get File (error off) → If has any value → Save to photo album → cleanup (a-Shell extension, коротка `rm` вкладається в ліміт) → сповіщення; Otherwise → алерт «не вийшло» з підказкою про `pip install -U yt-dlp`.

a-Shell **не підтримує x-callback-url** (перевірено по коду SceneDelegate/AppDelegate) — «Open X-Callback URL» завис би назавжди, тому саме `openurl shortcuts://run-shortcut`.

Команда в інтенті (v2.4; keepGoing=true — «продовжувати після помилки», щоб YTSave запустився і показав алерт навіть якщо yt-dlp упав):
```
cd ~Shortcuts
mkdir -p YTDown.nosync
cd YTDown.nosync
rm -f video.* *.part *.ytdl
yt-dlp --no-playlist --no-mtime -f "bv*[vcodec^=avc1][ext=mp4]+ba[ext=m4a]/b[ext=mp4]/b" --merge-output-format mp4 -o video.mp4 "[Вхід шортката]"
openurl shortcuts://run-shortcut?name=YTSave
```
Оскільки в інтенті openWindow=open команди виконуються послідовно і блокуюче, yt-dlp повністю завершується ДО рядка `openurl` — саме тому app-режим і потрібен.

- `-f "bv*[vcodec^=avc1]…+ba[ext=m4a]"` — найкраще H.264-відео + AAC-аудіо; yt-dlp склеює ffmpeg-ом без перекодування (секунди, без втрати якості).
- `--no-mtime` — інакше файл отримує дату публікації ролика і провалюється в галереї в минуле.
- `--no-playlist` — лінк з плейліста не тягне весь плейліст.
- URL у лапках — лінки з `&` без лапок ламають команду.
- `openurl` виконується навіть якщо yt-dlp упав — це свідомо: YTSave тоді не знайде файла (його стерто на старті) і покаже алерт про помилку.
- Імʼя YTSave — ASCII без пробілів свідомо: жодних проблем з подвійним URL-кодуванням у `run-shortcut?name=`.

## Налаштування на новому iPhone

1. Все з розділу «Налаштування» у [video-splitter.md](video-splitter.md) (a-Shell повний, закладка `~Shortcuts`).
2. Додатково в a-Shell один раз: `pip install yt-dlp`.
3. Імпортувати **обидва** шорткати: `Завантажити з YouTube 2.4.shortcut` і `YTSave.shortcut` (YTSave не чіпати і не перейменовувати — на нього посилається команда).
4. Перший запуск: роздати дозволи (відкриття a-Shell з шортката; у YTSave — Фото і a-Shell).

## Якщо перестало працювати

- Найчастіше: YouTube щось змінив → в a-Shell `pip install -U yt-dlp` (оновлення виходять кілька разів на місяць). Це перше, що треба пробувати; помилка yt-dlp видна прямо у вікні a-Shell.
- Алерт «Не вийшло» = yt-dlp завершився без файла; діагноз — у вікні a-Shell.
- Якщо після завантаження нічого не відбулось (YTSave не запустився) — перевірити, що шорткат YTSave існує і називається саме так.
- Вікові/приватні відео не скачаються — потрібні куки, у шорткат це не запечено свідомо.

## Граблі (додатково до [video-splitter.md](video-splitter.md))

- **a-Shell Execute Command в extension-режимі має ліміт часу** («The operation took too long to complete»): усе довше ~хвилини туди не влазить. Довгі команди — інтент з openWindow=open (в застосунок). Короткі (`rm`) — можна в extension.
- **`ashell://`-URL з шортката в шитері не працює**: Open URL з share-контексту блимає застосунком і викидає на домашній екран, команда не доїжджає. Для запуску команд із шитера — тільки інтент.
- `keepGoing` в інтенті a-Shell — це «Keep going after an error» (продовжувати після помилки), НЕ «не чекати завершення».
- a-Shell не має x-callback-url; зворотний звʼязок — тільки `openurl shortcuts://run-shortcut?name=…` останнім рядком команди.
- При запуску в застосунку (URL чи інтент openWindow=open) робоча директорія — груповий контейнер a-Shell, не `~`: перший рядок команди завжди `cd ~Shortcuts`.
- В a-Shell **немає `which`** (є яка-небудь незвична підмножина команд — перевіряти перед використанням). З keepGoing=false («зупинятись після помилки», v2.2) один невідомий рядок обірвав увесь скрипт до yt-dlp — виглядало як «нічого не сталося». Діагноз дав `log.txt`: обірвався одразу після рядка з `which`.
- **`>> файл 2>&1` ламає розбір перенаправлення в ios_system** (v2.3): парсер натрапляє на `2>&1` першим, ставить sharedErrorOutput і пропускає блок розбору `>>`, тому імʼя вихідного файлу зчитується як биткий `>` — весь вивід команди летить у нікуди, лог лишається порожнім (тільки те, що записав попередній `echo > log.txt`). Виглядає так, ніби yt-dlp мовчки нічого не зробив. Чисті варіанти в ios_system: `&> файл` (stdout+stderr в один файл, перезапис) або взагалі без перенаправлення (вивід на екран). У v2.4 лог прибрано — вивід на екрані a-Shell.

## Як зібрано

XML: scratchpad сесії 2026-07-20/21. Робочі фінали: `yt-downloader-v24.xml` (перший шорткат) + `ytsave-v10.xml` (YTSave), за шаблоном v2.6 різалки. Перший шорткат — єдина дія a-Shell Execute Command Intent (`openWindow=open`, `keepGoing=true`, `ShowWhenRun=false`), у команді токен «Вхід шортката» — `attachmentsByRange` `{211, 1}` (`Type: ExtensionInput`), зміщення перевірене скриптом (індекс U+FFFC). YTSave: Get File (error off) → If «has any value» (`WFCondition` 100, вхід — обгортка `{Type: Variable, Variable: {…}}`) → Save to camera roll + cleanup + notification; Otherwise → alert. Підпис: `plutil -convert binary1` → перейменувати в `.shortcut` → `shortcuts sign --mode anyone` (вхідний файл МУСИТЬ мати розширення `.shortcut`, з `.plist` буде «isn't in the correct format»).

Тупикові версії (не повторювати): v2.0/2.1 з `ashell://` URL (Text+URL Encode+Open URL, `yt-downloader-v20.xml`/`v21`) — не запускаються з шитера; v2.2 з `which` і keepGoing=false — обривались на невідомій команді; v2.3 з `>> log.txt 2>&1` — кривий парсер перенаправлень з'їдав вивід.
